## 📌 Project Overview

**Best Buy Phone** is a complete software engineering project for a smartphone e-commerce platform, developed following a **structured SE methodology**. The project covers two key phases of the software development lifecycle:

- **RAD** — Requirements Analysis Document: captures functional/non-functional requirements, use cases, domain model, and user interaction design (HCI)
- **SSD** — System Sequence Diagrams: models system behaviour, operation contracts, and interaction flows between actors and the system

The project demonstrates proficiency in **software analysis and design techniques** used in enterprise and agile software development processes, with particular attention to **user-centred design** (UOM — Interazione Uomo-Macchina).

- ✅ **Requirements elicitation** — functional and non-functional requirements analysis

## 📐 Document Breakdown

### RAD — Requirements Analysis Document

The RAD covers the **analysis phase** of the software lifecycle:

- **Problem Statement** — context and goals of the Best Buy Phone platform
- **Functional Requirements** — what the system must do (browsing, purchasing, user management, order tracking, etc.)
- **Non-Functional Requirements** — performance, usability, security, reliability constraints
- **Actors & Use Cases** — UML use case diagram with all system actors (Customer, Admin, etc.) and their interactions
- **Use Case Descriptions** — detailed flow of events for each use case (main success scenario + alternative flows)
- **Domain Model** — conceptual class diagram identifying domain entities and their relationships
- **HCI / Interaction Design** — user interface mockups and interaction flow design following human-computer interaction principles

### SSD — System Sequence Diagrams

The SSD covers the **design phase** entry point:

- **System Sequence Diagrams** — UML sequence diagrams showing interactions between external actors and the system as a black box for each use case
- **System Operations** — identification of all system-level operations triggered by actors
- **Operation Contracts** — formal pre/post-condition specifications defining system state changes

## 🔄 SE Methodology

The project follows the **Unified Process (UP)** / **OOAD** methodology:

```
Requirements Phase          Analysis Phase          Design Entry
       │                         │                      │
  Problem Statement         Domain Model            System Sequence
  Functional Req.           Use Case Detail         Diagrams (SSD)
  Non-Functional Req.       Operation flows         Operation Contracts
  Actors & Use Cases        HCI / UX design
       │                         │                      │
   └─────────── RAD Document ─────────────┘ SSD Document ─────┘
```
